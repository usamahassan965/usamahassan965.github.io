GradeScan — AI Grading of Scanned Answer Sheets

Turn a scanned PDF of exam answer sheets into structured grades: MCQ answers, handwritten free responses, and candidate numbers — with invalid and blank answers flagged automatically. Built for SEAMO (Southeast Asian Mathematical Olympiad, an international math competition) and hardened on a real 780-page scan batch. The MCQ lane runs fully offline with a two-stage YOLOv12 detector; only handwriting goes to a vision-language model.

02 Engineering Report — Three Versions

This system was rebuilt three times, and each rebuild was forced by reality: a new sheet format, messier scans, or a technique hitting its ceiling. The progression — from hardcoded pixel slicing, to adaptive contour detection, to trained object detectors — is the story of the project.

The task

Input: a scanned PDF where every page is one student's answer sheet. Output, per page: the selected option for every MCQ, the handwritten free-response answers, the handwritten candidate number, and the paper type. Two failure modes must be flagged, not guessed: IN (invalid) when a student marks more than one bubble (or writes more than 10 characters in a free-response row), and BL (blank) when nothing is marked. Getting IN/BL right is what separates a grader from a toy — a wrong guess silently changes a student's score.

Version 1 — Geometric OMR: prove the idea works

The first version treated bubble reading as pure geometry. Each page was split into three columns with np.array_split, the answer column sliced into 20 question rows and each row into choice cells at fixed pixel offsets, then binarized with an inverse threshold. The selected answer is simply the cell with the highest ink density — the percentage of non-zero pixels after thresholding. A marked bubble reads 22–53% ink; unmarked neighbours sit under 10%, so argmax is unambiguous. EasyOCR read the section header ("Questions 1 to 20") to switch between the two legacy layout variants automatically.

It worked — 20/20 on the sample sheets — but every coordinate was tuned to one specific PDF. The moment the client sent a new sheet design, everything broke. That was the trigger for version 2.

Version 2 — Contour detection + real-world robustness

Version 2 replaced every hardcoded coordinate with structure detection. The MCQ block is found by thresholding, morphological closing, and contour filtering — keep the contour whose aspect ratio says "tall vertical box" (h/w > 2) on the left half of the page. Inside each question row, a second contour pass finds the small answer boxes themselves, so ink percentage is measured inside the printed box, not in an arbitrary crop. All size filters became fractions of the page dimensions (e.g. width > page_width/8.27) instead of absolute pixels, so rescaled and low-resolution scans pass the same filters.

Two decision rules formalized the IN/BL flags from raw fill percentages:

  • Multi-choice (IN): two options whose fills are within 95% of each other and both clearly above the third → the student marked twice.
  • All-blank (BL): all fills within 98% of each other → nothing is marked, the tiny variation is print noise.

For crooked scans — tilted, upside-down, or shrunken pages — I benchmarked four deskew approaches (minAreaRect angle correction, two Hough-line estimators, and ocrmypdf). The homegrown methods each failed on some rotation class; ocrmypdf --deskew --rotate-pages fixed slant and 180° flips in one pass, so it became a preprocessing stage rather than custom geometry code.

Handwriting was the other half. I first trained a MobileNetV3-Small on EMNIST-balanced (47 character classes) as a fully offline reader — it reached 89.2% test accuracy, but single-character crops from real sheets broke it (a handwritten Y classified as W). Reading characters in isolation throws away context a vision-language model gets for free, so the free-response rows went to Gemini Flash at temperature 0 instead. The final version 2 pipeline merged candidate number + all five rows into one prompt per page with a strict output format — halving API calls and making the response parseable.

Version 3 — Two new formats, real scans, and the jump to YOLOv12

The final phase brought the Paper A–F and Paper K formats and, crucially, a real 780-page scan batch of student scripts. Five successive notebook iterations tried to stretch the contour approach to the new A–E bubble grid, and the notebooks record exactly where it died: the black divider lines between question rows merged adjacent contours, and real-scan noise (compression artifacts, faint print, slight misalignment) made the hand-tuned filters unreliable. Heuristics that are 98% right per box are wrong somewhere on nearly every 100-question page.

So detection became a learned problem. I labeled sheet crops in Roboflow and trained two YOLOv12 detectors:

I trained nano and medium variants of each and kept YOLOv12-m — the nano models matched mAP50 but were less stable on the localization-strict mAP50-95. The option detector's two classes are D (darkened) and ND (not darkened): detections are NMS-deduplicated with supervision, kept top-5 by confidence, then re-sorted left-to-right so index position maps directly to option letter A–E. Paper K reuses the same machinery with top_n=2 grid detections and a 10-row/5-row split decided by block height.

One refinement mattered more than any other: YOLO localizes, pixels decide. Trusting the D/ND labels alone mislabeled exactly the cases that matter most — multi-marked and blank rows. The shipped pipeline uses YOLO only to find each option box precisely, then computes ink fill inside the detected box: ≥2 boxes above 45% fill → IN, all below 45% → BL, otherwise argmax → the answer. The detector contributes localization robustness; the centuries-old "count the dark pixels" trick contributes the verdict.

Output and performance

Each page produces a JSON record; a page-level flag list routes uncertain pages to manual review. Timings from the real-batch run (Kaggle T4, free tier):

  • MCQ lane: ~0.8–1s grid detection + ~0.2s per question row after model warm-up — fully offline, so a 1,000-page batch costs nothing but GPU time (verified with generated 100- and 1,000-page stress-test PDFs).
  • Handwriting lane: ~4.7s per Gemini call — one call per page, and only this lane touches an API.
  • End-to-end: ~13.7s per page unoptimized and sequential; the lanes are independent, so parallelizing them and batching Gemini calls is the first easy win.

Before settling on the Gemini API, I evaluated Google Cloud Vision OCR and Vertex AI as handwriting readers — classic OCR mangled isolated handwritten characters, while the multimodal prompt approach could be told exactly what to look for and how to format it. The delivered system was demoed to the client as an API with screen-recorded walkthroughs.

03 Problems I Faced

Each of these changed the architecture. Problem first, then the solution that shipped.

1 · Hardcoded coordinates collapsed on every new sheet format

PROBLEM — Version 1 read bubbles by slicing the page at fixed pixel offsets. It scored 20/20 on the sample PDF and 0/20 on the next format the client sent: every offset pointed at the wrong region. Worse, the same format at a different scan resolution also broke, because absolute pixel thresholds (w > 200) don't survive rescaling.

SOLUTION — Stop assuming positions; detect structure. The MCQ block is found by its shape (a tall vertical contour on the left half), the answer boxes by theirs (small wide contours inside a row), and every size filter became a fraction of the page dimensions. The same code then handled full-size and shrunken scans without retuning.

2 · Tilted, flipped, and shrunken scans

PROBLEM — Real scan batches contain pages that are slanted, upside down, or scanned small. A tilted grid slices diagonally through bubble rows, and an upside-down sheet reads answers in reverse order — silently producing wrong grades rather than errors.

SOLUTION — I implemented three deskew methods by hand (minAreaRect angle recovery and two Hough-line angle estimators) and benchmarked them against ocrmypdf. Each manual method failed on some rotation class — minAreaRect can't tell 0° from 180°. ocrmypdf --deskew --rotate-pages handled slant and flips together (it uses OCR text confidence to decide orientation), so preprocessing became one battle-tested call instead of custom geometry code.

3 · Divider lines poisoned contour detection on the new grid

PROBLEM — The Paper A–F format prints a black divider line between question rows. When a row crop clipped that line, findContours merged the line with the option boxes into one blob — the box coordinates came back wrong, and fill percentages were measured over garbage regions. My notebook notes from the time: "Perfect method, but some rows have a black line under the boxes, so findContours mixes and doesn't pick the box clearly."

SOLUTION — Replace hand-tuned contour filters with a trained detector. A YOLOv12-m model labeled on Roboflow crops learned to localize option boxes despite the divider lines — lines are just context to a detector, not a failure mode. Detection mAP50 hit 0.995 and the whole class of "contour merged with a stray line" bugs disappeared.

4 · Clean synthetic samples vs. 780 pages of real scans

PROBLEM — Everything up to that point was validated on clean, digitally-generated sample sheets. The real batch — 780 compressed scanned student scripts — brought JPEG artifacts, faint printing, ink smudges, and slight per-page misalignment. Heuristics that were 98% accurate per box are wrong somewhere on nearly every page when a page has 100 boxes.

SOLUTION — Train on the distribution you deploy on. I labeled crops from the real scans themselves, retrained both detectors, and processed pages in batches of 10 to keep memory flat on a 780-page PDF. Pages the pipeline was unsure about were written to a flag list for manual review — 16 pages out of 780 — instead of being silently mis-graded.

5 · The detector alone got IN and BL wrong — the cases that matter most

PROBLEM — Reading answers directly from YOLO's D/ND labels worked for cleanly-marked rows but erred exactly on multi-marked and blank rows: a half-hearted pencil stroke would flip ND→D or vice versa. Those are the rows a grading system exists to catch — mislabeling IN as a normal answer changes a student's score.

SOLUTION — Split the job: YOLO localizes, pixels decide. The detector's only responsibility is finding each option box precisely; the verdict comes from ink-fill percentage inside the detected box with explicit thresholds — two or more boxes above 45% is IN, all below 45% is BL, otherwise the argmax is the answer. Deterministic, tunable, and explainable to the client one threshold at a time.

6 · Reading handwriting: a 89%-accurate CNN is still the wrong tool

PROBLEM — Free responses and candidate numbers are handwritten. I trained a MobileNetV3-Small on EMNIST (47 classes) to 89.2% test accuracy — and it still misread real crops (a clear Y predicted as W), because classifying characters in isolation drops all context, and 89% per character compounds to coin-flip odds on a 10-character candidate number.

SOLUTION — Use a vision-language model that reads the whole page in context. One Gemini 2.5 Flash call per page, temperature 0, with a strict-format prompt returning candidate number, paper type (A–F/K), and the five response rows — plus the business rule encoded right in the prompt: any row over 10 characters returns IN. The prompt's format contract makes the response machine-parseable, and merging everything into one call halved API cost.

04 What I'd Build Next

  • Parallel lanes + async Gemini batching — the MCQ and handwriting lanes are independent; running them concurrently and batching VLM calls should cut ~13.7s/page to under 5s.
  • Answer-key scoring & reports — join extracted answers against a key to produce per-candidate scores, and export CSV/Excel report cards.
  • Active-learning loop — flagged pages that a human corrects become new labeled training data, so the detectors improve on exactly their failure cases.
  • Local VLM for privacy — swap Gemini for a self-hosted Qwen2.5-VL so student data never leaves the machine — a hard requirement for many exam boards.
  • ONNX/TensorRT export — quantize the detectors for CPU-only deployment, removing the GPU requirement for the offline lane.
